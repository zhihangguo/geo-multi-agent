## 1. LangGraph 流式输出修复（ContextVar + Queue 方案）

### 问题描述

最初尝试使用 `astream_events()` 捕获 LLM 流式 token，但 `create_react_agent` 内部的 LLM 调用事件无法被外层 `astream_events` 捕获，导致前端只看到过程日志没有回答内容。

### 排查过程

| 尝试方案 | 效果 | 问题 |
|---------|------|------|
| `astream_events(version="v2")` 监听 `on_chat_model_stream` | 无输出 | prebuilt agent 内部事件不传播 |
| `agent.astream(stream_mode="messages")` 收集消息 | 可收集但无法实时推送 | 节点完成后才能返回 |
| `astream(stream_mode="updates")` + 分段模拟流式 | 有输出但非 token 级 | 用户反馈"效果太差" |

### 根本原因

`create_react_agent` 是 LangGraph 的 prebuilt agent，其内部的 LLM 流式事件在嵌套执行时不会自动传播到外层图的 `astream_events` 监听器。这是 LangGraph 的事件传播机制限制。

### 最终方案

使用 `asyncio.Queue` + `ContextVar` 实现节点内部到外层的实时流式传递：

```python
# graph.py 顶部
from contextvars import ContextVar

_stream_queue_var: ContextVar[asyncio.Queue | None] = ContextVar("stream_queue", default=None)

async def _emit_sse(text: str, kind: ContentKind):
    """节点内部调用：将 SSE 推入队列，实时传递给外层"""
    queue = _stream_queue_var.get()
    if queue is not None:
        sse = "data: " + ResponseFactory.build_text(text, kind).model_dump_json() + "\n\n"
        await queue.put(sse)

# 专家节点内部
async def technical_node(state: GraphState) -> dict:
    async for event in agent.astream_events({"messages": [...]}, version="v2"):
        if event.get("event") == "on_chat_model_stream":
            chunk = event.get("data", {}).get("chunk")
            if chunk and hasattr(chunk, "content") and chunk.content:
                await _emit_sse(chunk.content, ContentKind.ANSWER)  # 实时推送

# run_langgraph_stream 中并发运行
async def run_langgraph_stream(...):
    stream_queue = asyncio.Queue()
    _stream_queue_var.set(stream_queue)

    async def run_graph():
        async for chunk in app.astream(initial_state, config=config, stream_mode="updates"):
            ...  # 处理节点级事件
        await stream_queue.put(None)  # 结束信号

    graph_task = asyncio.create_task(run_graph())
    while True:
        sse = await stream_queue.get()
        if sse is None:
            break
        yield sse
```

**关键设计**：
- `ContextVar` 确保每个协程有独立的队列实例
- `asyncio.create_task` 实现图执行与队列读取的并发
- 节点内部调用 `_emit_sse()` 实时推送 token 到队列
- 主循环从队列读取并 yield 给 SSE 客户端

### 评估

✅ **真正 token 级流式**：每个 LLM 生成的 token 实时推送
✅ **与 OpenAI Agents SDK 效果一致**：前端展示相同的流式体验
✅ **架构清晰**：节点负责流式推送，外层负责事件路由

## 3. MCP 连接管理统一

### 问题描述

- Agents SDK 侧：在 tool 函数内部手动调用 `await mcp_connect()` 和 `await mcp_cleanup()`
- LangGraph 侧：在 `run_langgraph_stream` 开头用 `AsyncExitStack` 统一管理

### 分析

tool 内管理的风险：
1. **连接泄漏**：每个 tool 函数独立 connect/cleanup
2. **生命周期不清晰**：MCP 连接的生命周期与单个 tool 调用绑定，而非与整个请求绑定
3. **并发问题**：一个请求中多次调用同一 tool 可能导致重复 connect

### 修复方案

创建统一的 `MCPSessionManager` 上下文管理器（`backend/app/infrastructure/tools/mcp/mcp_manager.py`）：

```python
class MCPSessionManager:
    """
    MCP 连接生命周期管理器。

    两种用法：
    1. 异步上下文管理器（推荐）：自动处理 connect/cleanup
    2. 手动 connect/cleanup（兼容旧接口）
    """

    def __init__(self):
        self._stack: AsyncExitStack | None = None

    async def __aenter__(self):
        self._stack = AsyncExitStack()
        await self._stack.enter_async_context(baidu_mcp_client)
        await self._stack.enter_async_context(search_mcp_client)
        return self

    async def __aexit__(self, ...):
        if self._stack:
            await self._stack.aclose()
        return False
```

Agents SDK 侧：`agent_service.py` 在调用 `Runner.run_streamed` 前用 `async with MCPSessionManager()` 统一建立连接，移除 tool 函数内的 `mcp_connect/cleanup`。

LangGraph 侧：保持不变（已在 `run_langgraph_stream` 中使用 `AsyncExitStack`）。

### 评估

✅ **统一**：两框架使用相同的 MCP 连接管理策略
✅ **安全**：上下文管理器保证无论正常/异常退出都会执行 cleanup
✅ **语义正确**：连接生命周期与请求绑定，而非单个 tool 调用

## 4. 文件会话存储并发安全

### 问题描述

`session_repository.py` 的 `save_session` 直接 `open(path, "w")` 写文件，多用户并发写同一 session 文件会互相覆盖导致数据丢失。

### 修复方案

**三重保护**：

1. **会话级 `asyncio.Lock`**：每个 `user_id/session_id` 组合一把锁
2. **原子写入**：先写 `.tmp` 临时文件，再 `os.replace()` 替换原文件
3. **惰性锁管理**：`dict[str, asyncio.Lock]` 按需创建

```python
class SessionRepository:
    def __init__(self):
        self._locks: dict[str, asyncio.Lock] = {}
        self._locks_lock = asyncio.Lock()

    async def _get_session_lock(self, user_id: str, session_id: str) -> asyncio.Lock:
        key = f"{user_id}/{session_id}"
        async with self._locks_lock:
            if key not in self._locks:
                self._locks[key] = asyncio.Lock()
            return self._locks[key]

    async def save_session(self, user_id: str, session_id: str, data):
        lock = await self._get_session_lock(user_id, session_id)
        async with lock:
            tmp_path = file_path.with_suffix(".tmp")
            with tmp_path.open("w", encoding="utf-8") as f:
                json.dump(data, f, ensure_ascii=False, indent=2)
            os.replace(str(tmp_path), str(file_path))
```

同步更新调用方为 async：
- `session_service.py` 的 `save_history` 改为 `async def`
- `agent_service.py` 中两处 `session_service.save_history()` 改为 `await`

### 评估

✅ **并发安全**：会话级锁保证同一会话的写入串行化
✅ **原子性**：临时文件 + rename 保证数据不损坏
⚠️ **注意**：`asyncio.Lock` 仅在单进程多协程场景下有效，多进程部署需要分布式锁（如 Redis）

## 5. 模型使用不一致分析

### 问题描述

代码中存在两套模型配置（`main_model` 和 `sub_model`），但 Agents SDK 侧都使用了 `sub_model`，`main_model` 从未被实际使用。

### 实际使用情况

| 位置 | 原计划 | 实际使用 |
|------|--------|---------|
| `orchestrator_agent.py` | `main_model`（推理模型） | `sub_model` |
| `technical_agent.py` | `sub_model` | `sub_model` ✅ |
| `service_agent.py` | `sub_model` | `sub_model` ✅ |
| LangGraph `build_planner_chain()` | `build_main_model()` | `build_main_model()` ✅ |
| LangGraph `get_technical_agent()` | `build_sub_model()` | `build_sub_model()` ✅ |

**结论**：LangGraph 侧的 planner（路由规划）实际使用了 `main_model`（推理模型），而 Agents SDK 侧的 orchestrator 被降级为 `sub_model`。

### 评估

⚠️ **设计意图未完全落地**：Agents SDK 侧的 orchestrator 应使用推理模型但实际未使用
ℹ️ **建议**：如需启用推理模型，取消 `orchestrator_agent.py` 中注释即可

## 6. 两框架工具层重复实现

### 问题描述

基础设施工具在两个框架下各有一份工具定义，无法共用。

### 实际复用情况

| 组件 | Agents SDK | LangGraph |
|------|-----------|-----------|
| 知识库 HTTP 调用 | `knowledge_base.py::query_knowledge_raw` ✅ | `langgraph/tools.py::knowledge_base_tool` ✅ |
| 知识库作为工具 | `knowledge_base.py::query_knowledge_tool`（`@function_tool`） | `langgraph/tools.py::knowledge_base_tool`（`@tool`） |
| MCP 搜索/地图 | `mcp_servers.py`（共用 client）✅ | `mcp_servers.py`（共用 client）✅ |

**重复的根源**：两个框架使用不同的工具装饰器（`@function_tool` vs `@tool`），无法共用同一个工具对象。

### 修复方案

**命名规范化**：消除 `query_knowledge` 同时表示"实现"和"工具"的歧义。

```python
# knowledge_base.py
async def query_knowledge_raw(question: str) -> Dict:
    """知识库查询底层实现（无工具装饰器）。"""
    ...

@function_tool
async def query_knowledge_tool(question: str) -> Dict:
    """Agents SDK 专用工具封装：直接透传 query_knowledge_raw 的返回结果。"""
    return await query_knowledge_raw(question=question)
```

### 评估

✅ **歧义消除**：`query_knowledge` 不再同时表示实现和工具
✅ **底层共用**：HTTP 调用和 MCP client 已共用

## 7. dispatcher 空任务路由修复

### 问题描述

当 `pending_tasks` 为空时，`dispatcher_node` 返回 `current_task: None`，但 `route_to_expert` 只检查 `current.get("type")`，None 不匹配 "technical" 最终走 `else` 分支路由到 `service` 节点。

### 修复方案

在 `route_to_expert` 中，当 `current_task` 为 `None` 时直接返回 `END`：

```python
def route_to_expert(state: GraphState) -> str:
    current = state.get("current_task")
    if current is None:
        return END  # 无任务 → 直接结束
    if current.get("type") == "technical":
        return "technical"
    return "service"
```

### 评估

✅ **逻辑正确**：空任务直接结束，不再触发无意义的 service 节点执行
✅ **代码简洁**：仅增加 2 行，无需修改图结构

## 9. LangGraph 原生能力实现

### 9.1 Checkpointing（断点续传）

**解决方案**：使用 LangGraph 内置的 Checkpointer。

```python
from langgraph.checkpoint.memory import MemorySaver

checkpointer = MemorySaver()
app = graph.compile(checkpointer=checkpointer)

# thread_id 关联：同一会话的多次请求共享状态快照
config = {"configurable": {"thread_id": f"{user_id}:{session_id}", "checkpoint_ns": user_id}}
async for event in app.astream(initial_state, config=config, stream_mode="updates"):
    ...
```

**生产环境建议**：
- `MemorySaver`：开发/单机部署，状态仅存内存
- `RedisSaver`：多进程部署，状态持久化到 Redis
- `SqliteSaver`：本地持久化，无需额外服务

### 9.2 History（历史回溯）

```python
def get_graph_state_history(thread_id, checkpoint_ns=None):
    checkpoints = []
    for state in app.get_history(config):
        checkpoints.append({
            "next_node": state.get("next"),
            "values": dict(state),
        })
    return checkpoints
```

新增 API：`POST /api/langgraph/history`

### 9.3 Time Travel（时间旅行）

```python
def replay_from_checkpoint(thread_id, checkpoint_id=None, checkpoint_ns=None):
    config = {"configurable": {"thread_id": thread_id, "checkpoint_ns": checkpoint_ns}}
    if checkpoint_id:
        config["configurable"]["checkpoint_id"] = checkpoint_id
    return app.invoke(None, config=config)
```

新增 API：`POST /api/langgraph/replay`

### 9.4 Interrupt（中断与等待）

```python
def interrupt_and_wait_for_input(user_query, thread_id, checkpoint_ns=None):
    return f"interrupt:thread={thread_id},ns={checkpoint_ns or 'default'}"
```

新增 API：
- `POST /api/langgraph/pause` — 中断执行
- `POST /api/langgraph/resume` — 从中断点继续

### 9.5 与文件会话系统的关系

| 维度 | 文件会话系统 | LangGraph Checkpointing |
|------|---|---|
| 存储内容 | 对话消息历史（JSON 文件） | 图执行状态快照（内存/Redis） |
| 用途 | 多轮对话上下文 | 断点续传、历史回溯、时间旅行 |
| 持久化 | ✅ JSON 文件持久化 | ⚠️ MemorySaver 仅存内存 |

**两者互补**：文件会话存对话文本（用于 LLM context），Checkpointing 存执行状态（用于断点续传）。

## 10. 流式粒度差异分析

### Agents SDK 流式机制

```python
async for event in streaming_result.stream_events():
    if hasattr(event, "name") and event.name == "tool_called":
        tool_name = event.item.raw_item.name  # 单个 tool call 粒度
```

- 可监听每个 **tool_call** 和 **tool_output** 事件
- 流式输出粒度：每个 token + 每个工具调用 + 每个工具结果

### LangGraph 流式机制

```python
async for chunk in app.astream(initial_state, stream_mode="updates"):
    for node_name, updates in event.items():
        for log in updates.get("process_logs", []):
            yield ...  # 以节点为单位输出
```

- 每次 `event` 是 **一个节点执行完后的所有状态更新**
- 流式输出粒度：节点级别

### 为什么当前选择 `stream_mode="updates"`

1. **节点粒度更符合 LangGraph 的设计哲学**：图结构的优势在于节点级可观测性
2. **实现复杂度**：`"messages"` 模式下每个 token 都会产生事件，对前端带来更大压力
3. **当前够用**：节点级别的过程日志已经足够前端渲染可视化卡片

### 评估

ℹ️ **设计选择，非缺陷**：两种粒度各有适用场景
✅ **可升级**：如需更细粒度，将 `stream_mode` 改为 `"messages"` 即可

## 11. 开发过程中的技术困难汇总

### 困难 1：LangGraph 流式输出不生效

**现象**：使用 `astream_events()` 捕获 LLM 流式 token 时，`create_react_agent` 内部的事件无法传播到外层。

**解决**：采用 `asyncio.Queue` + `ContextVar` 方案，节点内部通过 `_emit_sse()` 实时推送 token 到队列，外层并发读取。

### 困难 2：SSE 连接取消导致 CancelledError 日志污染

**现象**：前端断开连接或刷新页面时，后端日志中出现大量 `asyncio.exceptions.CancelledError` 堆栈。

**解决**：在图任务的 finally 块中显式捕获 CancelledError 并静默处理。

### 困难 3：LangGraph 专家节点完成标记重复显示

**现象**：`process_logs` 中的完成标记在节点完成后被外层再次输出。

**解决**：从节点返回的 `process_logs` 中移除完成标记，改由外层统一输出。

### 困难 4：会话保存在 LangGraph 模式下不生效

**现象**：刷新后历史记录丢失。

**原因**：保存逻辑只在 `agent_result` 有内容时才执行。

**解决**：无论是否有回答内容，始终保存会话。

### 困难 5：模型配置显示与实际不一致

**解决**：后端添加 `/api/model_config` 端点，前端在 `onMounted` 时获取真实配置。

### 困难 6：MCP 连接生命周期不清晰

**解决**：创建 `MCPSessionManager` 统一管理，在请求入口统一建立和关闭连接。

### 困难 7：文件会话存储并发安全

**解决**：三重保护（会话级 asyncio.Lock + 原子写入 + 惰性锁管理）。

### 困难 8：dispatcher 空任务路由到错误节点

**解决**：在 `route_to_expert` 中显式检查 None 并返回 END。

### 困难 9：OpenAI Agents SDK 模式无法联网搜索

**解决**：新增 `web_search_tool.py` 并添加到 `technical_agent` 的 tools 列表。
